# Task: PapiflyFX Settings Refactor (Phase 2: Composable UI Refactor)

## 👤 Role & Ownership
- **Lead Agent:** `@ops-engineer` (assisted by `@feature-dev`)
- **Primary Modules:** `papiflyfx-docking-settings`, `papiflyfx-docking-settings-api`
- **Objective:** Eliminate hardcoded settings structures and introduce a composable, definition-driven form infrastructure (Workstream A).

## 📚 Reference Context
- **Implementation Plan:** `spec/papiflyfx-docking-settings/2026-04-10-review0/plan.md`
- **Accepted Architecture:** `spec/papiflyfx-docking-settings/2026-04-10-review0/refactor-plan-gemini-total.md`
- **Current Progress:** `spec/papiflyfx-docking-settings/2026-04-10-review0/progress.md` (Confirm Phase 1 completion)

## 🧱 Key Invariants
1. **Definition-Driven:** UI must be generated compositionally from `SettingDefinition` and standard metadata. Manual form code should be minimized.
2. **Deterministic State:** Use observable properties for dirty and valid tracking. Eliminate all polling/timer-based refresh.
3. **No FXML:** Follow the repository guideline—all UI must be programmatic JavaFX.
4. **Theme Alignment:** Use `-pf-ui-*` tokens and shared CSS classes from `ui-common.css`.

## 🚀 Initial Implementation Tasks (Phase 2)

### 1. Shared Form Binder (Workstream A.1)
- Introduce a shared form binder/presenter around `SettingControl` implementations.
- Ensure the binder can generate a UI layout dynamically based on a collection of `SettingDefinition`.

### 2. Category Refactor (Workstream A.2)
- Refactor built-in categories (Appearance, Network, AI Models, MCP Servers) to use the new `SettingControlFactory` and declarative definitions.
- Target: drastic reduction in manual form building code in these categories.

### 3. Validation & State Aggregation (Workstream A.3)
- Centralize validation aggregation for the active category.
- Replace ad-hoc dirty tracking with observable state (Property/Binding).
- Ensure the "Apply" button enablement is bound directly to this observable state.

## 🛡️ Review Gates (Mandatory)
Obtain sign-off from the following specialists:
- **@ui-ux-designer:** Audit the new form layouts, spacing, and CSS token usage.
- **@feature-dev:** Review the refactoring of content-related categories (AI, MCP).
- **@core-architect:** Ensure the new binder doesn't introduce circular dependencies between API and implementation.
- **@qa-engineer:** Review UI tests for form generation, validation feedback, and "Apply" state.

## 🧪 Validation Strategy
- **Headless Build:** `./mvnw clean compile`
- **Module Tests:** `./mvnw -pl papiflyfx-docking-settings -am test`
- **Manual Verification:**
    - Verify that adding a new `SettingDefinition` to a category automatically renders it in the UI.
    - Verify that the "Apply" button reacts instantly to input changes (validation/dirty state) without a timer.
    - Check that theme switching applies to the new composable forms correctly.
