# Task: PapiflyFX Settings Refactor (Phase 1: Runtime & Scope Safety)

## 👤 Role & Ownership
- **Lead Agent:** `@ops-engineer`
- **Primary Modules:** `papiflyfx-docking-settings`, `papiflyfx-docking-settings-api`
- **Objective:** Consolidate `SettingsRuntime` ownership and align scope semantics across categories.

## 📚 Reference Context
- **Implementation Plan:** `spec/papiflyfx-docking-settings/2026-04-10-review0/plan.md`
- **Accepted Architecture:** `spec/papiflyfx-docking-settings/2026-04-10-review0/refactor-plan-gemini-total.md`
- **Current Progress:** `spec/papiflyfx-docking-settings/2026-04-10-review0/progress.md`

## 🧱 Key Invariants
1. **Single Runtime:** One application instance must own exactly one `SettingsRuntime`. Session restore and `ServiceLoader` paths must NOT silently create a second runtime.
2. **Truthful Scopes:** The UI toolbar must only display scopes (Application, Workspace, Session) that are actively supported by the current settings category.
3. **Explicit Injection:** Refactor components to depend on host-injected runtimes rather than self-bootstrapping.

## 🚀 Initial Implementation Tasks (Phase 1)

### 1. Runtime Consolidation (Workstream B)
- Refactor `SettingsStateAdapter` (in `papiflyfx-docking-settings`) and `DefaultSettingsServicesProvider` (in `papiflyfx-docking-settings-api`) to remove hidden default runtime creation.
- Ensure all restore paths fail closed or require explicit injection.
- Verify that login and docking restore paths inject and reuse the same instance.

### 2. Scope Cleanup (Workstream C)
- Classify all built-in categories (Appearance, Network, AI, etc.) by their supported scope policy.
- Remove hard-coded scope usage in favor of `SettingDefinition.scope()`.
- Refactor `SettingsPanel` toolbar to dynamically show/hide scope options based on the active category.
