## 🧱 Key Invariants
1. **No Secret Re-exposure:** The UI must never reload or display stored secret values. Administration must use aliases and lifecycle actions (replace/rotate/clear).
2. **Atomic Persistence:** Setting writes must use an "atomic swap" (write to temp file then move) to prevent file corruption during power loss or crashes.
3. **Resilient Recovery:** The system must have a documented and implemented strategy for recovering from corrupted setting files.

## 🚀 Initial Implementation Tasks (Phase 3)

### 1. Security Category Redesign (Workstream D.1 & D.2)
- Redesign `SecurityCategory` to eliminate the reloading of stored secret values into UI controls.
- Implement lifecycle actions: Rotate/Replace (set new value) and Delete/Clear (remove existing value).
- Ensure the UI reflects whether a secret is "Set" or "Not Set" without revealing the content.

### 2. Atomic Save Behavior (Workstream D.3)
- Implement atomic save logic for JSON setting files.
- Process: Write to `.tmp` file -> Verify write -> Atomic move/rename to target file.

### 3. Corruption Recovery & Threat Model (Workstream D.4 & D.5)
- Define and implement a corruption recovery strategy (e.g., fallback to `.bak` or reset to defaults).
- Revisit and document the encrypted-file fallback threat model, specifically addressing machine-metadata key derivation.
